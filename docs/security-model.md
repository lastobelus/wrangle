# Security Model

## Trust boundaries

`wrangle` has three relevant trust boundaries:

1. The caller of `wrangle`
2. The selected backend executable or transport
3. The filesystem and environment visible to the current user

`wrangle` does not sandbox the backend. If a backend is malicious or compromised, it should be assumed capable of acting with the same local access available to the current user and process environment.

## Trusted vs untrusted inputs

Trusted inputs:

- direct CLI invocation by the local user
- explicitly selected backend binaries
- explicit prompt files passed by the local user

Less-trusted or externally supplied inputs:

- newline-delimited task specs for parallel execution
- backend stdout/stderr streams
- resumed session identifiers supplied by another system

## Current controls

- prompt content is not logged verbatim by default
- stdout JSON parsing is line-size bounded
- stderr capture is truncated to a fixed maximum
- retained normalized events are capped
- task-spec prompt files are disabled unless explicitly allowed
- environment inheritance is reduced by default

## Parallel task-spec trust model

Parallel execution introduces a new input channel: newline-delimited JSON task specs read from stdin. This section documents how `wrangle` treats those inputs.

### Task-spec fields and trust classification

| Field | Trust level | Notes |
|-------|-------------|-------|
| `id` | Untrusted | Validated: non-empty, no duplicates across the graph |
| `task` | Trusted by caller, untrusted by `wrangle` | Passed through to the backend as-is. `wrangle` does not interpret or sanitize task content. |
| `workDir` | Untrusted | If present, used as provided for that task's working directory; otherwise the base config `work_dir` is used. |
| `dependencies` | Untrusted | Validated: all referenced ids must exist in the graph, no self-refs, no cycles |
| `sessionId` | Untrusted | Validated: alphanumeric/dash/underscore, max 128 chars |
| `backend` | Untrusted | Validated against known backend names |
| `model` | Untrusted | Passed through to the backend |
| `agent` | Untrusted | Resolved against agent config |
| `promptFile` | Restricted | Disabled by default. Must be explicitly enabled via `--allow-task-prompt-files` or `allow_task_prompt_files: true` in `RuntimeConfig`. When disabled, any task spec that includes `promptFile` fails immediately with `TaskPromptFilesDisabled`. |
| `permissionPolicy` | Untrusted | Validated against the selected backend's supported policies |
| `transportMode` | Untrusted | Validated against the selected backend's supported modes |

### Prompt file handling

Prompt files are a sensitive input because they allow a task spec to reference an arbitrary filesystem path whose contents are passed to the backend. This is why prompt files from task specs are **disabled by default**:

- A single-task CLI invocation with `--prompt-file` is treated as a trusted local-user action.
- A parallel task spec with `promptFile` is treated as a less-trusted input because it arrives programmatically (often generated by another system).
- Callers must opt in with `--allow-task-prompt-files` / `allow_task_prompt_files: true` to use prompt files in parallel task specs. This opt-in is intentional and should only be enabled when the caller controls or trusts the source of all task specs.

### Graph validation guarantees

Before any work is spawned, `wrangle` validates the full task graph:

1. **Structural validation**: non-empty ids, non-empty task fields, no duplicate ids
2. **Reference validation**: all dependency references point to existing task ids; self-references are rejected
3. **Cycle detection**: DFS-based cycle detection identifies the exact cycle path (e.g., `a -> b -> c -> a`) so callers can diagnose and fix the graph
4. **Backend capability validation**: every task's resolved backend, transport mode, and permission policy are checked against the backend's advertised capabilities

These checks run in the validation phase before any backend process is spawned. A validation failure produces a precise, actionable error identifying the offending task and field.

### Input validation error taxonomy

| Error | Trigger | Diagnostic |
|-------|---------|------------|
| `EmptyTaskId` | A task spec has an empty `id` field | "parallel task id must be non-empty" |
| `EmptyTask` | A task spec has an empty `task` field | "parallel task must have a non-empty 'task' field (id='...')" |
| `DuplicateTaskId` | Two tasks share the same `id` | "duplicate parallel task id: ..." |
| `SelfDependency` | A task lists itself in `dependencies` | "parallel task cannot depend on itself: ..." |
| `UnknownDependency` | A task references a non-existent id | "task '...' depends on unknown task id '...'" |
| `CircularDependency` | The dependency graph contains a cycle | "circular dependency detected: a -> b -> c -> a" |
| `TaskPromptFilesDisabled` | A task spec includes `promptFile` without opt-in | "prompt files from task specs are disabled without --allow-task-prompt-files" |
| `UnsupportedPermissionPolicy` | A task's backend doesn't support the requested policy | "backend '...' does not support permission policy '...'" |
| `UnsupportedTransport` | A task's backend doesn't support the requested transport | "backend '...' does not support transport mode '...'" |

## Permission policy

`wrangle` models permissions through `PermissionPolicy`, then maps that model to backend-specific flags:

- `Default`: backend default approval behavior — no flag applied, backend decides
- `Ask`: explicitly request interactive approval before each action (reserved; not yet mapped to any backend flag)
- `Auto`: semi-automatic mode — the backend proceeds with safe operations and asks before destructive ones
- `Bypass`: request full automation mode where the backend supports it

Not every backend supports every policy. Capability reporting (`available_backends()`, `wrangle backends`) lists the supported policies per backend so callers can decide before execution. Unsupported combinations produce a clear `UnsupportedPermissionPolicy` error rather than silently degrading. `Ask` is not currently advertised for any backend because no backend has a distinct flag mapping for it; using `Ask` will result in a policy-not-supported error.

### Backend policy support

| Backend | Default | Ask | Auto | Bypass |
|---------|---------|-----|------|--------|
| Codex   | yes     | —   | yes  | yes    |
| Claude  | yes     | —   | —    | yes    |
| Gemini  | yes     | —   | —    | yes    |
| Qwen    | yes     | —   | —    | yes    |
| Opencode| yes     | —   | —    | —      |

`Bypass` is only advertised where there is a defensible backend-native full-auto equivalent. `Ask` is not advertised because it currently maps to no distinct backend flag. `Auto` is only advertised where the semantics are clear enough to explain and test.

## Persistent transports

Persistent transports increase the security surface because state may outlive one request. That affects:

- session attachment
- cleanup behavior
- credential lifetime
- whether execution context survives across requests

For that reason, `wrangle` keeps transport identity explicit:

- `OneShotProcess`
- `PersistentBackend`
- `WrangleServer` (reserved)

V1 implements only `OneShotProcess`. Opencode persistent transport is a planned v2 addition, and a `wrangle`-native server is a planned v3 addition.
